# 런처 혜니팩 검색·설치 + 토큰 저장소 설계

- **작성일**: 2026-07-10
- **대상**: `cloudflare-worker`(공개 API + 관리 패널) + `apps/launcher`(Tauri Rust + 렌더러) — HyeniAPIServer는 후속(§8)
- **상태**: 브레인스토밍 완료, 사용자 확정 사항 반영 → 스펙 리뷰 대기

## 1. 목적과 문제

### 목적
1. 런처에서 혜니팩을 **검색해서 설치** — 지금은 관리자가 `.hyenipack` 파일을 사용자에게 직접 전달해야 함.
2. **신규 사용자 원클릭 온보딩** — 디스코드 `/인증` 한 번으로 인증+팩 설치+토큰 설정까지.
3. **기간제 서버 수명주기 지원** — 서버 기간 종료 시 해당 혜니팩을 비공개로 전환(목록·조회·다운로드 차단).

### 현재 구조의 문제
- **닭-달걀**: 팩 다운로드에 /인증 토큰이 필요한데, 토큰은 프로필 game_dir의 `config/hyenihelper-config.json`에만 저장되고, 딥링크 인증(`apply_auth`)은 "HyeniHelper가 설치된 기존 프로필"에만 기록한다(MODE1·2 모두 `has_hyenihelper` 조건). **프로필 0개면 /인증 자체가 실패** → 최초 설치 경로가 성립 안 함.
- **토큰 오이식 버그(잠복)**: `pack_apply_update`([pack.rs](../../apps/launcher/src-tauri/src/pack.rs))가 MS 계정 access_token을 워커에 보냄. 워커 `isValidToken`은 TOKEN_CHECK_API_URL(디스코드 /인증 화이트리스트)로 검증하므로 401 예상. 워커모드 쪽 동일 버그는 이미 수정된 이력 있음(hyeni.rs 주석). **실 테스트 전 — 이번에 함께 수정.**
- **공개 팩 목록 API 없음**: 목록은 admin(Access 뒤)에만 존재.

### 핵심 제약 (사용자 확인, 2026-07-10)
**토큰은 (디스코드 채널 × 마인크래프트 서버) 세트별 발급**(HyeniAPIServer). 같은 서버라도 채널이 다르면 다른 토큰. → **단일 전역 토큰으로 프로필 config를 덮으면 안 됨.** 프로필 config 기록은 언제나 서버 매칭을 통과한 토큰만.

### 핵심 관찰 (설계의 근거)
1. **워커 다운로드는 토큰 스코프를 안 봄** — `isValidToken`은 `{token}`만 보내 유효성/화이트리스트만 확인. 스코프가 실제로 중요한 곳은 **게임 내 인증(프로필 config)** 뿐.
2. **딥링크에 스코프가 실려 옴** — `hyenimc://auth?token=...&servers=...`.
3. **id로 다운로드→설치 파이프는 이미 있음** — `pack_apply_update` = latest 조회→`download_pack_version`→`hyenipack_import`. 최초 설치는 이 조합에 "id 선택 + 프로필 생성"만 얹으면 됨.
4. 런처 팩 업데이트 확인은 **404를 "업데이트 없음"으로 우아하게 처리**(실행 차단 없음) → 비공개 전환이 기존 유저를 깨지 않음.

## 2. 설계 — 계층별

### 2-1. 런처 토큰 저장소 (다중 엔트리, 서버 스코프별)

```
런처 DB: hyeni_tokens = [ { token, servers: [...], received_at }, ... ]
```

- **딥링크 수신 시**: 기존 MODE1/2 프로필 기록은 그대로 수행하고, **추가로 저장소에 (servers → token) 엔트리 적재**. 같은 servers 세트가 이미 있으면 최신 토큰으로 교체(현행 MODE1 "무조건 덮어쓰기"와 동일 의미). servers 없는 MODE2 딥링크는 스코프 없는 엔트리로 저장(다운로드 폴백에만 사용, 프로필 매칭에는 불사용).
- **프로필 0개여도 성공**: 기존 "프로필을 찾을 수 없습니다" 에러 → "인증되었습니다. 혜니팩 설치 시 자동 적용됩니다." 성공 처리로 변경.
- 저장소는 화면/세션과 무관한 영구 저장. 진짜 유효성 판정은 서버측(화이트리스트) — 저장소는 보관함.

#### 토큰 사용 우선순위 (모든 다운로드 경로 공통)
```
1순위: 프로필 config 토큰 (game_dir/config/hyenihelper-config.json)
2순위: 런처 저장소 토큰 (폴백; 다운로드는 스코프 무관이라 안전)
둘 다 없음: "Discord /인증 필요" 안내
```
| 경로 | 동작 |
|---|---|
| 팩 업데이트(기존 프로필) | 프로필 토큰 → 저장소 폴백. **MS 토큰 오이식 교정** |
| 팩 최초 설치(프로필 없음) | 저장소 토큰 |
| 모드 다운로드(실행 전 게이트·수동 패널) | 프로필 토큰 → 저장소 폴백 |

#### config 기록 규칙 (스코프 보호)
- **프로필 config에 토큰을 쓰는 것은 언제나 서버 매칭 통과 시만** (`servers_dat_contains` 재사용).
- 팩 설치 완료 시: 팩이 깔아준 `servers.dat`와 저장소 엔트리의 `servers`를 매칭 → 맞는 토큰만 config에 기록. 매칭 없으면 기록하지 않고 "이 서버의 디스코드 채널에서 /인증" 안내(그 딥링크가 기존 MODE1로 정확히 기록).
- 실행 전 게이트에서 config가 비어 폴백할 때: 저장소에 **그 프로필 서버와 매칭되는** 엔트리가 있으면 config에 기록 후 사용(다운로드+게임 인증 동시 해결). 매칭 없으면 **현행대로 "/인증 필요" 중단** — 모드만 받아지고 접속은 안 되는 어중간한 상태 방지.

### 2-2. 워커 — 공개 팩 목록 + 공개/비공개

#### 팩 메타: `modpacks/{id}/meta.json` (신규)
```json
{ "name": "시즌3 혜니팩", "hidden": false,
  "minecraft": { "version": "1.21.1", "loaderType": "neoforge", "loaderVersion": "21.1.186" },
  "updatedAt": "<iso>" }
```
- **latest.json과 분리된 별도 객체** — latest.json은 게시/롤백마다 덮어써지므로 플래그를 거기 두면 게시 한 번에 비공개가 풀리는 사고가 남. meta.json은 게시/롤백/버전삭제와 독립.
- `name`·`minecraft`: 게시 시 `.hyenipack`의 `hyenipack.json`에서 추출해 저장(이미 admin이 lenient unzip 보유) → 공개 목록 API가 **매 요청 unzip 없이** 이름·환경 제공(이름 검색 재료). 게시마다 최신 값으로 갱신하되 `hidden`은 보존. 롤백 시에도 대상 버전 기준으로 갱신.
- 기존 팩(meta 없음): `hidden=false` 취급, name은 id 폴백, minecraft는 null. 다음 게시 때 자연 백필.

#### 공개 API
- **`GET /api/v2/modpacks`** (신규, 완전 공개 — 사용자 확정): hidden 아닌 팩만.
  ```json
  { "packs": [ { "id", "name", "latestVersion", "breaking",
                 "minecraft": { "version", "loaderType", "loaderVersion" } | null } ] }
  ```
  응답은 R2 prefix 스캔 + 각 팩의 `latest.json`(버전·breaking) + `meta.json`(name·minecraft·hidden)만으로 구성 — unzip 없음. 서버측 `?q=` 검색은 두지 않음 — 목록+런처 클라이언트측 필터가 검색 역할(§6).
- **비공개(hidden=true)의 효과 = id 전체 비공개** (사용자 확정): 목록 미등장 + `GET /{id}/latest`·`/{id}/versions`·`/download/...` 모두 404. 기존 설치 유저는 실행 유지·업데이트만 중단(§1 관찰 4).
- 관리 API(Access 뒤)는 hidden 무관하게 항상 접근 가능.

#### 관리 API/UI
- `PATCH /admin/api/modpacks/{id}/visibility` body `{hidden: bool}`.
- admin 목록 응답에 `hidden` 포함. UI: 팩 상세(또는 rail)에 공개/비공개 토글 + 비공개 배지.

### 2-3. 런처 Rust 커맨드

- `pack_list_available()` → `GET /api/v2/modpacks` (공개, 토큰 불필요).
- `pack_install_from_worker(profile_id, pack_id)` → latest 조회 → 토큰(우선순위 규칙) → `download_pack_version` → `hyenipack_import` 재사용 → **설치 후 서버 매칭 토큰 config 기록**(2-1 규칙). 프로필 생성 자체는 렌더러가 기존 `profile_create` 후 이 커맨드를 호출(팩이 game_version/loader를 덮으므로 생성 시 초기값은 임의).
- `pack_apply_update` 토큰 교정(2-1 표).
- 토큰 저장소 CRUD는 hyenimc-core(DB) + 딥링크 핸들러(hyeni.rs) 수정.

### 2-4. 렌더러 — 진입점 UI (사용자 확정: 새 프로필 생성 통합)

- 새 프로필 만들기 플로우에 **"혜니팩으로 시작"** 선택지 → 팩 목록 + **검색창(클라이언트측 필터: 이름/id)** → 선택 → (필요 시 인증 안내) → 프로필 생성+설치 진행 표시 → 완료.
- 저장소·config에 유효 토큰이 하나도 없으면 목록은 보여주되 설치 시점에 "Discord에서 /인증 후 다시 시도" 안내.
- 설치 완료 시 토큰 매칭 결과 표시: 자동 적용됨 / "이 서버의 채널에서 /인증 필요".

### 2-5. 딥링크 확장 — `hyenipack` 파라미터 (런처 측 선제 준비)

- `hyenimc://auth?token=...&servers=...&hyenipack={packId}` — `parse_auth_url` 확장(파라미터 없으면 기존 동작 그대로, 하위호환).
- 처리 순서: 인증 처리(저장소 적재+프로필 기록) → 팩 조회(`/latest`) → **확인 다이얼로그 1개**("'{name}' v{version}을 설치할까요?") → 승인 시 새 프로필 생성+설치+토큰 기록(2-3/2-1 재사용).
  - 딥링크는 외부에서 만들어질 수 있는 입력 → 무확인 설치 금지. 팩 출처는 어차피 우리 워커뿐이라 위험은 낮지만 확인은 유지.
  - 이미 같은 팩(modpack_id)이 설치된 프로필이 있으면: 설치 대신 "이미 설치됨 → 업데이트 확인" 안내.
  - 팩이 비공개/부재(404)면: 인증만 처리하고 팩 안내는 생략(에러 아님).
- **HyeniAPIServer(디스코드 봇) 변경은 후속·독립**(§8) — 런처 먼저 배포 가능.

## 3. 시나리오 검증

| 시나리오 | 동작 |
|---|---|
| 신규(프로필 0): 채널A /인증 → 팩A 설치 | 저장소 적재 → 목록/다운로드 OK → servers.dat 매칭으로 토큰 자동 기록. 한 번에 완료 |
| 신규: /인증 없이 팩 설치 시도 | 목록은 보임(공개) → 설치 시 "/인증 필요" 안내 → 인증 후 재시도 |
| 딥링크에 hyenipack= 포함(신규) | 인증+확인 다이얼로그+설치+토큰까지 원클릭 온보딩 |
| 서버 A·B 프로필 공존, 채널B 재인증 | (B→토큰B) 엔트리만 추가/교체. A 프로필 config 불변(매칭 기반) |
| 같은 서버·다른 채널 재인증 | servers 동일 → 최신 토큰으로 교체(현행 MODE1과 동일 의미) |
| 인증한 서버와 다른 서버의 팩 설치 | 설치는 됨(다운로드 스코프 무관), config 미기록 + "해당 채널에서 /인증" 안내 → MODE1이 마무리 |
| 기간제 종료 → 비공개 토글 | 목록·latest·다운로드 404. 기존 유저 실행 유지, 업데이트만 중단 |
| 기존 프로필(config에 토큰 있음) | **아무 변화 없음** — config 토큰 1순위 |

## 4. 하위호환 / 마이그레이션
- 딥링크: `hyenipack` 파라미터 없으면 기존 동작. 프로필 매칭 0건의 의미만 에러→성공(저장)으로 변경.
- meta.json 없는 기존 팩: hidden=false·name=id 폴백, 다음 게시 때 백필.
- 공개 API 기존 라우트(latest/versions/download)는 형식 불변 — hidden 체크만 추가.
- 기존 프로필·기존 인증 흐름(MODE1/2): 불변.

## 5. 보안 노트
- 토큰을 런처 DB에 저장 — 이미 game_dir 평문 JSON에 놓이는 자격증명과 동등 수준. 신규 노출 없음.
- 유효 토큰이면 다른 서버의 공개 팩도 다운로드 가능 — 현행 모드 다운로드와 동일한 신뢰 수준(신규 노출 아님). 팩별 다운로드 스코프 검증은 비목표(§6).
- 딥링크 자동 설치는 확인 다이얼로그로 게이트. 팩 콘텐츠 출처는 관리자 게시물뿐.

## 6. 비목표 (YAGNI)
- 서버측 검색 쿼리(`?q=`) — 팩 수십 개까지는 목록+클라 필터로 충분. 수백 규모가 되면 그때 추가.
- 팩별 다운로드 스코프 검증(API 서버 확장) — 기간제 통제는 비공개 토글로 충분.
- MS 계정 검증으로의 전환.
- 모드 의존성(dependencies) 해석 — 별도 과제(포맷 정의부터).

## 7. 확정된 결정 (2026-07-10 사용자)
- 검색은 필요(팩 증가 전제) — 단, 서버는 목록 API, 검색창은 런처 클라 필터로.
- 목록 API는 **완전 공개**(다운로드만 토큰).
- 진입점은 **새 프로필 생성 플로우 통합**.
- 비공개 = **id 전체 비공개**(목록+조회+다운로드), 관리 UI 토글.
- 토큰은 채널×서버 세트별 → **단일 전역 토큰 금지**, 다중 엔트리 저장소 + config 기록은 서버 매칭 시만, config 토큰 1순위.
- 딥링크 `hyenipack` 파라미터: **런처 측 지금 준비**, API 서버(봇)는 후속.
- 설치 직전 확인 다이얼로그 유지(무확인 설치 금지).

## 8. 후속 작업 (이 스펙 범위 밖)
- HyeniAPIServer/디스코드 봇: `/인증` 딥링크에 `hyenipack={id}` 부가(채널별 매핑). 런처 배포 후 아무 때나.
